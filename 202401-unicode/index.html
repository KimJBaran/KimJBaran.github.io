Common Unicode encoding pitfalls

Jan 2, 2025

Unicode is a character encoding standard that’s widely used. It maps bytes to letters, numbers, emoji, etc., for us humans to read. I’m sure there are still exceptions where you’ll find other character encodings, but for anything web, mobile, or desktop related you’ll be dealing with Unicode. At a surface level this makes it look like you’ve to never think about character encodings ever again. After all, everything is Unicode now, right? The deceptive simplicity of Unicode makes it easy to internationalize your web-site or app, but there are plenty of edge cases that you’ll need to be aware of.

What are character encodings? Character encodings map bytes to characters. ASCII is a great example for encoding English: it maps 95 printable characters (a-z, A-Z, 0-9, punctuation, etc.) and has mappings for 33 control codes (new line, tab, etc.). That’s 128 encoded mappings which would fit into 7 bits, but you’ll most likely only ever encounter ASCII in 8 bits — one byte per character with the most significant bit set to zero. Allocating a single byte per character worked fine for English texts back in the day, but that isn’t nearly enough anymore in any international context.

Unicode can encode more than 1.1 million characters. Have a look at Unicode Planes and try to figure out how many characters it could theoretically encode, if you feel too jolly today. Since 1.1 million characters can’t be encoded by a single byte, Unicode comes in three major representations:

  • UTF-8: uses a single byte for ASCII and up to three bytes for international characters (variable width),
  • UTF-16: uses two bytes for the most common international characters, emoji, symbols, etc., but can double to four bytes when needed (variable width),
  • UTF-32: always uses four bytes to encode a character (fixed width), and
  • a few more representations that are rare. Have a look at UTF-EBCDIC, if you fancy punch cards.

Before diving in deeper, I want to point out two things that I won’t address here:

  • Unicode’s quirk that allows some character to be represented by different code points (for example, “ê” can be both U+00EA and U+0065 U+0302), and
  • endianness that handles whether the most-significant or least-significant byte is stored first.

The first issue may cause your app to return empty search results even though the “right” characters were entered by your users. It just happens that their internal representation in Unicode doesn’t match. The second issue is a problem when loading/saving data across different operating systems or hardware architectures. Getting endianness wrong will result in entirely wrong characters to be displayed. While these are big problems, there are simpler Unicode pitfalls to watch out for. Here I’ll look at:

  • string length, and
  • random access of individual characters within a string.

Let’s start with a simple case that revolves around the character ”💖” which is the Unicode scalar U+1F496. Since I’ve copied this from the Swift documentation let’s see how long that single character string is in Swift.

% swift repl
  1> let s = "💖"
s: String = "💖"
  2> s.count
$R0: Int = 1

Nice! Swift deals with Unicode strings rather well, so there isn’t any surprise here. Swift 5 switched from UTF-16 to UTF-8 and it perfectly handles this character that’s three bytes when encoded in UTF-8. If you’re dealing with Swift a lot, then you’ll notice that string operations typically involve an iterator which makes dealing with multi-byte characters possible. You lose random access of characters though.

How does JavaScript handle this? Even though we’re writing HTML in UTF-8, JavaScript is dealing with strings in UTF-16. This needs to keep us on our toes immediately, because you’ll find blogs that claim that JavaScript is UTF-8 too. It isn’t. It’s complicated, but ECMA262 4.3.17 calls out UTF-16 specifically. JavaScript is also very old and it has to be backwards compatible, so here’s a bit of a surprise where the heart character appears to “grow” into two characters:

% node
> const s = "💖";
undefined
> s.length
2
> s[0]
'\ud83d'
> s[1]
'\udc96'
> [...s]
[ '💖' ]
> [...s].length
1

JavaScript doesn’t deal with Unicode well unless you’re using iterators (see my use of spread syntax above). You might be tempted to say that this is obvious and that random access naturally breaks when dealing with anything but UTF-32. Or does it? Let’s have a look at Python:

% python3
>>> from sys import getsizeof
>>> s = "💖"
>>> len(s)
1
>>> s[0]
'💖'
>>> getsizeof(s)
64
>>> s = "💖💖"
>>> getsizeof(s)
68

Python returns the correct string length and it also allows random access of characters within the string. I used getsizeof here to figure out which character encoding Python internally uses. It’s UTF-32 (look at the four byte increment when adding a second heart). UTF-32 isn’t Python’s default character encoding though. It appears that Python dynamically changes the encoding depending on the text it needs to represent. Quite some thought went into their string handling, because Python’s documentation also highlights and offers solutions for comparing Unicode strings — a problem that I’ve said I’d gloss over here. Like Swift, Python is rather well suited to deal with Unicode, but it still requires attention from developers to do The Right Thing.

Takeaways:

  • string length might not count characters in the string
  • random access to characters in a string may not work
  • string/character comparison needs special handling
  • serialization/de-serialization across platforms is difficult

I hope that this post shines some light on immediate issues that arise when dealing with Unicode. These issues are minor in comparison to the myriad of different individual character encodings that we had to deal with before Unicode, but these are still issues must be addressed.
